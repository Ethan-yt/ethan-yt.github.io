Retrofit优雅封装

Posted by Yan Tan on 2017-03-31

安卓的Retrofit是一个优秀的网络封装库，和Rxjava结合能发挥更好的优势，虽然Retrofit已经封装了大量的代码，可是结合自己的实际使用时，往往还会重复写下许多代码。通过一套优雅的封装，可以为以后每次请求省下不少代码。

总的来说，通过封装，实现了以下几点

  1. 统一添加Headers，实现OAuth 2.0请求
  2. 在接口中不必出现ApiResponse这种包装，直接填写数据类型
  3. 统一对HTTP错误和HTTP200时业务逻辑中的错误进行处理在onError中处理
  4. 用户没有登录时提供跳转到登录Activity的方式
  5. 用户Token过期时自动使用refreshToken刷新，刷新后重新获取上个请求，全程对用户透明

虽然代码比较多，需要增加几个类，可是封装后，每次HTTP请求只需要这样就可以了：

单例模式：ApiHelper

首先，我们需要对我们自己的接口进行封装。在我的项目中，我定义了一个接口名为ApiService，用来实现项目中API的具体业务逻辑，也就是retrofit.create()产生的Service。由于这个对象需要在每次进行HTTP请求时都要访问，所以这里使用单例模式，将它封装起来，为一个静态变量，在任何地方都可以得到访问，省下了许多初始化的代码。代码如下：

通过静态内部类：ApiServiceHolder 保存单例，可以保证线程安全。static语句中的代码可以在外部类访问内部类成员时执行，这就正好实现了懒加载。所以我们将初始化代码放在static内。

在初始化代码中，做了这几件事：

  1. 建立一个OkHttpClient对象，禁止他的重定向（我自己的项目要求，重定向时直接抛出HTTP异常302，可以不设置）
  2. 为OkHttpClient添加拦截器，这样就可以统一添加Headers了。由于我的项目中采用OAuth 2.0，每次访问网络请求基本都需要Token，所以必须添加一个统一的Headers，这就需要拦截器的帮助了。
  3. 使用上一步的OkHttpClient建立一个Retrofit对象，增加一个转换器，以便处理错误异常，包括HTTP错误以及和后端约定好的业务流程上的错误。
  4. 使用上一步的Retrofit建立一个Service，即单例模式中的单例，我们要在外部访问的对象。

因此，以后可以使用getInstance()方法获取Service的实例了。

在此之上，可以将ApiService替换成泛型进行进一步封装，从而使一个类得到不同的Service对象。由于我的项目中暂时用一个ApiService就可以了，所以我没有用泛型。

另外，每次请求都需要写这么一大长串，是不优雅的。所以我们要进一步的封装它。

Token过期的刷新

经过ExceptionEngine的处理，我们得到了一个ApiException。之后就可以处理Token过期自动刷新的逻辑了。

这里直接返回一个Observable作为flatMap的结果，再传送至retryWhen（如果代码不记得了往前翻一下）。

在这里要实现的逻辑是：

  1. 如果刷新成功则设定新token继续原来的请求（使用onNext 触发retry）
  2. 失败则再试一次（使用onNext 触发retry）
  3. 试了几次还不行就说明refreshToken也过期了，只能提示用户重新登录（使用onCompleted终止retry，不能用onError阻止）

触发onNext很简单，情况（1），获取新token成功时直接触发onNext，情况（2）使用onErrorResumeNext将一切错误转换成Observable.just(new Token())，这里和情况（1）做一个区别，传入的token是空的。至于触发OnCompleted，不能简单的返回一个Observable.empty，因为原来的序列是一个Observable<Throwable>，现在通过flatMap将每一个throwable转换成一个Observable，存在一个内部和外部的关系，由于某些设计上的原因，内部的onCompleted是不会被外部接收的。所以，我们唯一的做法就是，先进入onNext，再根据onNext的数据判断是否放弃这个数据。一旦放弃这个数据，剩下的就是onCompleted了。

takeWhen操作符会将onNext的数据传入，经过一系列判断，假如放弃这个数据，直接返回false。反之返回true。

对于情况（1），刷新token成功，会得到一个非空的Token。所以直接设定新token继续原来的请求。

对于情况（2），刷新token失败，会得到一个空的Token。

对于情况（3），就是重试了3次还不行。什么情况会大于3次呢？首先用户提交一个请求，这个请求没有其他问题就只提示token失效，之后刷新token失败，失败，失败，就到这里了。多次失败有可能有2个原因：1、用户突然这时候断网了，2、用户的refreshToken也失效了。3、用户刷新的token是假token。出现这种情况就让用户重新登录一次吧。
